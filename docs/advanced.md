## Включение поддержки автономного режима
В Hydejack v8 представлена ​​экспериментальная поддержка автономного режима "кэширование по мере использования". Это реализовано с помощью... [Service Worker API](https://developer.mozilla.org/en-US/docs/Web/API/Service_Worker_API), Новый браузерный стандарт, который теперь поддерживается в последних версиях всех основных браузеров. Однако это очень мощная функция, и её следует использовать с большой осторожностью.

Для включения этой функции ваш контент должен соответствовать следующим критериям:
* Содержимое не меняется между развертываниями (например, ручное добавление элементов в `_site` и т. д.)
* Все ресурсы в `assets` являются неизменяемыми, т. е. они никогда не меняются (при изменении файла в `assets` ему необходимо присвоить новое имя, и ссылки должны указывать на новый файл).

* Сайт в основном самодостаточен, т. е. ресурсы обслуживаются с одного домена (поддержка офлайн-режима по умолчанию не будет загружать ресурсы с внешних сайтов)
* Сайт обслуживается по протоколу HTTPS (это требование Service Worker)

Чтобы включить эту функцию, создайте файл `sw.js` в корне вашего проекта и добавьте следующее содержимое:
{% raw %}
```js
---
---
importScripts("{{ '/assets/js/sw.js' | relative_url }}?t={{ site.time | date_to_xmlschema }}");
```
{% endraw %}

Это загрузит основной скрипт сервис-воркера из ресурсов Hydejack.`site.time` Эта часть необходима для того, чтобы сервис-воркер каждый раз создавал новую сборку вашего сайта, что запускает обновление, и делал его "разным по байтам".

В файл `config.yml` в разделе `hydejack` добавьте следующее:

```yml
offline:
  enabled: true
  cache_version: 1
```

Текущая реализация не кэширует ресурсы с внешних доменов. Нет способа узнать, соответствуют ли внешние сайты указанным выше условиям, поэтому кэширование может быть проблематичным и приводить к неожиданному поведению.

Например, Google Analytics использует GET-запросы для отправки данных о просмотрах страниц, каждый из которых без этой политики кэшировался бы сервис-воркером. Часто обновляемые изображения, такие как значки, никогда бы не изменились.

![Gem Version](https://badge.fury.io/rb/jekyll-theme-hydejack.svg)

Однако, если вы включаете ресурсы, размещенные на другом домене и не изменяющиеся, вы можете добавить параметр запроса `sw-cache` к URL-адресу, например:

    https://upload.wikimedia.org/wikipedia/commons/b/b1/57_Chevy_210.jpg?sw-cache

Это приведет к тому, что они будут кэшироваться как ресурсы из папки `assets`.

Если вы хотите предоставлять доступ к файлу из папки `assets`, но НЕ кэшировать его для использования в автономном режиме, добавьте вместо этого параметр запроса `no-cache`:

    /assets/lfs/download.bin?no-cache


![57 Chevy](https://upload.wikimedia.org/wikipedia/commons/b/b1/57_Chevy_210.jpg?sw-cache)


### Как работает автономное хранилище

Пользовательская реализация сервис-воркера Hydejack хранит файлы для автономного использования на трех разных уровнях:

Оболочка
: Файлы оболочки — это основные файлы Hydejack (CSS, JS), которые изменяются только между обновлениями версий.
Если вы внесли изменения в какой-либо из этих файлов после включения поддержки автономного режима, вам необходимо принудительно обновить их, увеличив значение `cache_version` в файле конфигурации.

Ресурсы
: *Предполагается, что они неизменяемы.* Другими словами, каждый файл кэшируется неограниченно долго. Например: если вы хотите обновить изображение после включения поддержки автономного режима, добавьте изображение под другим именем и измените ссылку в содержимом. В качестве альтернативы вы можете увеличить значение `cache_version`, но это удалит все остальные кэшированные файлы из кэша ресурсов.

Содержимое
: Кэш содержимого использует тот факт, что ваше содержимое не может изменяться между сборками, поэтому оно может храниться для автономного использования до тех пор, пока вы не загрузите новую сборку. На данный момент весь кэш контента очищается каждый раз при публикации нового контента (в будущих версиях кэширование может осуществляться на основе дат последнего изменения).

Также следует отметить, что реализация всегда будет кэшировать страницы, перечисленные в разделе «Юридическая информация», а также страницу «404.html», которая будет отображаться, когда пользователь находится в автономном режиме.


## Добавление пользовательской иконки социальной сети
Hydejack по умолчанию включает в себя ряд иконок социальных сетей (фактически, все, что предоставляет [IcoMoon](https://icomoon.io/)), но поскольку ситуация постоянно меняется, вполне вероятно, что важная для вас платформа в какой-то момент может отсутствовать.

Вы можете добавить любую платформу, просто указав полный URL-адрес. Однако будет использоваться резервная иконка <span class="icon-link"></span>.

{:.note}

### Создание шрифта иконок
Чтобы добавить пользовательскую иконку социальной сети, вам необходимо использовать [приложение IcoMoon](https://icomoon.io/app/) (бесплатное) для создания пользовательского веб-шрифта иконок. Однако важно, чтобы сгенерированный шрифт включал все иконки, уже используемые Hydejack. Для этого найдите файл `selection.json` в [`assets/icomoon/selection.json`](https://github.com/hydecorp/hydejack/blob/v6/assets/icomoon/selection.json) и загрузите его в приложение через функцию "Импорт иконок".
Затем используйте приложение для добавления ваших иконок.
Для получения дополнительной помощи обратитесь к [документации IcoMoon](https://icomoon.io/#docs).

После того, как вы создали и загрузили шрифт иконок из IconMoon, полностью замените папку `icomoon` в `assets`. Имейте в виду, что будущие обновления Hydejack заменят эту папку.

### Добавление метаданных платформы
На втором шаге необходимо добавить метаданные сети в `_data/social.yml`.
Запись выглядит следующим образом:

~~~yml
deviantart:
  name: DeviantArt
  icon: icon-deviantart
  prepend: "https://"
  append: ".deviantart.com"
~~~

`name`
: The name of the network. Used for the title attribute and screen readers.

`icon`
: The icon CSS class. Can be chosen during the IcoMoon creation process.

`prepend`
: Optional. A string that is prepended to the username to form the link to the profile. If the final URL should be `https://<username>.deviantart.com`, this would be `https://`

`append`
: Optional. A string that is appended to the username to form the link to the profile. If the final URL should be `https://<username>.deviantart.com`, this would be `.deviantart.com`.


## Как организован CSS в Hydejack
Hydejack использует довольно уникальный подход к CSS, который обусловлен возможностью
встраивать основные правила CSS в тег `style` в `<head/>` страницы (для увеличения скорости загрузки),
в то время как остальные правила хранятся в отдельном файле.

Стили написаны на SCSS и находятся в папке `_sass`, которая выглядит следующим образом:

~~~
├── hydejack
│   ├── __inline__
│   ├── __link__
│   ├── _base.pre.scss
│   ├── ...
│   └── _social.pre.scss
├── pooleparty
│   ├── __inline__
│   ├── __link__
│   ├── _base.pre.scss
│   ├── ...
│   └── _type.pre.scss
├── mixins.scss
├── my-inline.scss
├── my-style.scss
├── syntax.scss
└── variables.scss
~~~

Правила стиля организованы вместе с компонентами (точнее, темами), такими как «боковая панель» и «нижний колонтитул».
Кроме того, существуют два отдельных фреймворка, «pooleparty» и «hydejack»,
которые выросли из оригинальных проектов [Poole](http://getpoole.com/) и [Hyde](http://hyde.getpoole.com/).
Poole/party содержит более общие правила стиля, в то время как Hyde/jack содержит правила, более специфичные для данной темы.
Однако со временем это разделение размылось.

Внутри этих папок вы заметите папки `__inline__` и `__link__`.
Неудобные названия даны намеренно, поскольку их содержимое генерируется скриптом и не должно изменяться напрямую.
Исходные файлы находятся в той же папке и заканчиваются на `.pre.scss`.
Это полностью валидные SCSS-файлы, но содержат комментарии, указывающие, какие строки должны быть встроены, а какие должны загружаться асинхронно.

Правила следующие:
* Каждая строка между `// <<< inline ` и `// >>>` будет встроена.
* Каждая строка между `// <<< link ` и `// >>>` будет связана.
* Каждая строка, не содержащаяся в блоке и заканчивающаяся `// inline`, будет встроена.
* Каждая строка, не содержащаяся в блоке и заканчивающаяся `// link`, будет связана.
* Каждая строка, для которой ни одно из вышеперечисленных условий не выполняется, будет включена в оба варианта.

Фактическое разделение происходит с помощью скрипта `.scripts/build-css.sh` (требуется Node.js 8+).
Вы можете запустить скрипт один раз, используя

~~~bash
$ npm run build:css
~~~

или перестраивать CSS при каждом изменении файла

~~~bash
$ npm run watch:css
~~~

Обратите внимание, что файлы `my-inline.scss` и `my-style.scss` не затрагиваются этим.
Также, поскольку все файлы являются допустимыми SCSS-файлами, разделение является полностью необязательным.
Если вы хотите создать только один обычный CSS-файл, добавьте

```yml
hydejack:
  no_inline_css: true
```

в ваш конфигурационный файл.
